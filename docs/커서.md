# 🛠 Cursor Rule 설정 가이드 (Project Instructions)

이 문서는 **AI 사주 Master** 프로젝트의 절대 원칙(Absolute Rules)을 정의합니다.  
Cursor는 코드 생성 시 본 문서의 규칙을 최우선으로 준수해야 하며, 명시되지 않은 인증 로직이나 아키텍처를 임의로 추가해서는 안 됩니다.

---

## 1. 식별자 및 인증 규칙 (User Identity)

### 1.1 No Auth Flow
- 회원가입(Signup), 로그인(Login), 이메일 인증(Verification)과 관련된 로직을 **절대 생성하지 마라**.
- OAuth, JWT, Session 기반 인증 등 어떠한 인증 메커니즘도 구현하지 않는다.
- 사용자는 이메일 입력만으로 즉시 서비스를 이용할 수 있어야 한다.

### 1.2 Email as Primary Key
- `users` 테이블의 **Primary Key는 `email`**이다.
- 모든 사용자 식별은 `user_id`가 아닌 **이메일 문자열 매칭**으로 처리한다.
- 이메일은 시스템 전반에서 **유일 식별자(Unique Identifier)**로 사용된다.

### 1.3 Silent User Creation (Upsert)
- 사용자가 이메일을 입력하고 서비스를 호출할 때:
  1. `users` 테이블에서 해당 이메일을 조회한다.
  2. 존재하지 않을 경우 즉시 **생성(Create)** 한다.
  3. 존재할 경우 기존 레코드를 **재사용(Get)** 한다.
- 이 과정에서 추가 정보 입력이나 인증 절차를 요구하지 않는다.

**예시 SQL**
```sql
INSERT INTO users (email)
VALUES (:email)
ON DUPLICATE KEY UPDATE last_login = CURRENT_TIMESTAMP;
```

---

## 2. DB 및 관계 설계 (Database)

### 2.1 Users 테이블
```sql
CREATE TABLE users (
    email VARCHAR(100) PRIMARY KEY,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    last_login TIMESTAMP NULL
);
```

### 2.2 Saju Profiles 테이블
```sql
CREATE TABLE saju_profiles (
    profile_id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_email VARCHAR(100) NOT NULL,
    nickname VARCHAR(50) NOT NULL,
    birth_date DATE NOT NULL,
    birth_time VARCHAR(10) NOT NULL DEFAULT 'UNKNOWN',
    gender ENUM('MALE', 'FEMALE') NOT NULL,
    year_ganji VARCHAR(10),
    month_ganji VARCHAR(10),
    day_ganji VARCHAR(10),
    time_ganji VARCHAR(10),
    is_primary BOOLEAN NOT NULL DEFAULT FALSE,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_email) REFERENCES users(email)
);
```

### 2.3 FK Mapping
- `saju_profiles.user_email`은 반드시 `users.email`을 참조해야 한다.
- 관계는 **1:N (users → saju_profiles)** 구조를 따른다.

### 2.4 Data Type Rules
- `birth_time`은 `VARCHAR(10)` 타입이며 기본값은 `'UNKNOWN'`이다.
- 성별은 반드시 `ENUM('MALE', 'FEMALE')` 또는 대문자 문자열로 처리한다.
- 모든 날짜 및 시간 계산은 **한국 표준시(KST, UTC+9)** 기준으로 수행한다.

### 2.5 Caching Strategy
- 만세력 엔진 호출 결과는 다음 컬럼에 **Upsert** 방식으로 저장해야 한다.
  - `year_ganji`
  - `month_ganji`
  - `day_ganji`
  - `time_ganji`
- 이를 통해 동일한 요청에 대한 **중복 계산을 방지**하고 성능을 향상시킨다.

---

## 3. 비즈니스 로직 및 보정 (Domain Logic)

### 3.1 Time Normalization
- `birth_time`이 `'UNKNOWN'`인 경우:
  - 애플리케이션 레벨에서 `is_time_unknown = True`로 간주한다.
  - 만세력 엔진에 전달할 때는 내부적으로 **`12:00:00`(정오)**로 보정하여 처리한다.
  - 계산된 시주는 `time_ganji` 컬럼에 저장된다.

### 3.2 Engine Separation
- **AI가 사주를 직접 계산하게 하지 마라.**
- 반드시 백엔드의 만세력 엔진 함수 호출 결과만 AI 해석 프롬프트에 전달해야 한다.
- 모든 간지 및 오행 데이터는 서버에서 계산된 값만 사용한다.

### 3.3 Timezone Standard
- 모든 생년월일 및 생시 계산은 **한국 표준시(KST, UTC+9)** 기준으로 처리한다.
- 타임존 변환이 필요한 경우 서버에서 선처리 후 만세력 엔진에 전달한다.

---

## 4. AI 아키텍처 (Function Calling)

### 4.1 Function Calling Only
- OpenAI의 **Function Calling** 기능을 사용하여 AI와 백엔드 간 상호작용을 구현한다.
- 다음 API를 도구(Tool)로 등록해야 한다.
  - `calculate_saju`
  - `get_user_history`

### 4.2 No RAG
- 고전 문헌 검색을 위한 **RAG(Retrieval-Augmented Generation)** 기능은 구현하지 않는다.
- 벡터 데이터베이스, 임베딩 검색 등은 본 프로젝트의 범위에 포함되지 않는다.

### 4.3 Function Response Standard
- 모든 함수는 **JSON 형식의 구조화된 데이터**를 반환해야 한다.
- 반환 데이터에는 최소한 다음 정보가 포함되어야 한다.
  - `year_ganji`
  - `month_ganji`
  - `day_ganji`
  - `time_ganji`
  - `five_elements` (오행 분포)

### 4.4 Function Definition Example
```json
{
  "name": "calculate_saju",
  "description": "사용자의 생년월일과 생시를 기반으로 사주 팔자를 계산한다.",
  "parameters": {
    "type": "object",
    "properties": {
      "birth_date": {
        "type": "string",
        "format": "date"
      },
      "birth_time": {
        "type": "string",
        "description": "HH:MM 형식 또는 'UNKNOWN'"
      },
      "gender": {
        "type": "string",
        "enum": ["MALE", "FEMALE"]
      },
      "calendar_type": {
        "type": "string",
        "enum": ["solar", "lunar"]
      }
    },
    "required": ["birth_date", "birth_time", "gender", "calendar_type"]
  }
}
```

### 4.5 Interpretative Guidance
- `birth_time = 'UNKNOWN'`인 경우, AI 응답의 서두에 다음 안내 문구를 반드시 포함해야 한다.

> **"태어난 시간을 모르시는 경우의 해석입니다."**

- 이 문구는 시스템 프롬프트 또는 응답 생성 로직에서 강제 적용되어야 한다.

---

## 5. API 설계 원칙 (FastAPI)

### 5.1 Backend Framework
- 백엔드는 **FastAPI**를 사용하여 구현한다.
- 모든 API는 RESTful 원칙을 따르며, OpenAPI(Swagger) 문서를 자동으로 생성해야 한다.

### 5.2 Recommended Endpoints
| Method | Endpoint | Description |
|-------|---------|-------------|
| POST | `/saju/calculate` | 사주 계산 및 프로필 생성 |
| GET | `/users/{email}/profiles` | 사용자 프로필 목록 조회 |
| GET | `/users/{email}/history` | 사용자 해석 이력 조회 |

### 5.3 Error Handling
- 모든 API는 일관된 JSON 에러 응답 형식을 사용해야 한다.
```json
{
  "error": {
    "code": "INVALID_INPUT",
    "message": "입력값이 올바르지 않습니다."
  }
}
```

---

## 6. 보안 및 개인정보 보호 (Security & Privacy)

- 이메일은 사용자 식별을 위한 최소한의 정보로만 수집한다.
- 모든 API 통신은 **HTTPS**를 사용해야 한다.
- 불필요한 개인정보는 저장하지 않는다.
- 사용자의 요청이 있을 경우 관련 데이터를 삭제할 수 있어야 한다.
- 로그에는 민감한 개인정보를 포함하지 않는다.

---

## 7. 확장성 고려 사항 (Future Extensibility)

- 향후 이메일 인증 기반 로그인 시스템으로 확장 가능하도록 설계한다.
- 소셜 로그인(Google, Apple 등) 연동을 고려할 수 있다.
- 결제 및 구독 모델 추가가 가능하도록 구조를 유지한다.
- 푸시 알림을 위한 `push_subscriptions` 테이블 확장을 고려한다.

---

## 8. Cursor를 위한 절대 금지 사항 (Strict Prohibitions)

Cursor는 다음 기능을 **절대 생성해서는 안 된다.**

1. 회원가입(Signup) 또는 로그인(Login) 기능
2. 이메일 인증(Verification) 로직
3. JWT, Session, OAuth 등 인증/인가 시스템
4. RAG 또는 벡터 데이터베이스 기반 검색 기능
5. AI가 직접 사주를 계산하는 로직
6. `user_id` 기반의 사용자 식별 방식

# 🐘 코끼리 아가씨 프로젝트 절대 준수 원칙 (Project Rules)
> **작성일:** 2026-04-09
> **대상:** 프로젝트 내 모든 백엔드 로직 및 AI 아키텍처 구현

---

## 1. 식별자 및 인증 규칙 (User Identity)
- **No Auth Flow:** 회원가입(Signup), 로그인(Login), 이메일 인증(Verification) 로직을 절대 생성하지 마라.
- **Email as PK:** `users` 테이블의 PK는 `email`이다. 모든 사용자 식별은 이메일 문자열 매칭으로 처리한다.
- **Silent User Creation:** 사용자가 이메일을 입력하고 서비스를 호출할 때, DB에 해당 이메일이 없으면 즉시 생성(Create)하고 있으면 조회(Get)한다.

## 2. DB 및 관계 설계 (Database)
- **FK Mapping:** `saju_profiles`의 `user_email`은 `users.email`을 참조한다.
- **Data Type:** - `birth_time`은 `VARCHAR(10)`이며 기본값은 `'UNKNOWN'`이다.
    - 성별은 반드시 `ENUM('MALE', 'FEMALE')` 또는 대문자 문자열로 처리한다.
- **Caching:** 만세력 엔진 호출 결과(`year_ganji` 등)는 반드시 `saju_profiles` 테이블에 저장(Upsert)하여 중복 계산을 방지하라.

## 3. 비즈니스 로직 및 보정 (Domain Logic)
- **Time Normalization:** `birth_time`이 `'UNKNOWN'`인 경우, 만세력 엔진에 넘길 때는 내부적으로 `12:00:00`으로 보정하여 처리하라.
- **Engine Separation:** AI가 사주를 직접 계산하게 하지 마라. 반드시 백엔드의 만세력 엔진 함수를 호출한 결과값만 AI 해석 프롬프트에 전달하라.

## 4. AI 아키텍처 (Function Calling)
- **No RAG:** 고전 문헌 검색(RAG) 기능을 구현하지 마라.
- **Function Calling Only:** OpenAI의 Function Calling 기능을 사용하여 `calculate_saju`, `get_user_history` 등의 API를 도구로 등록하라.
- **Interpretative Guidance:** `is_time_unknown`이 `True`인 프로필의 경우, AI 답변 서두에 "태어난 시간을 모르시는 경우의 해석입니다"라는 안내를 포함하도록 시스템 프롬프트를 구성하라.
